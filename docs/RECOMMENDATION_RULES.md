# Recommendation Rules

## Principle

Recommendations must be generated by code from approved policies and verified input data.

LLMs must not decide:

- Which policy is eligible
- How much money is available
- Whether duplicate receipt is allowed
- Which combination is optimal

## Inputs

Recommendation calculation consumes:

- Approved policy versions
- Rule-engine eligibility output
- Code-calculated support amounts
- Duplicate-benefit compatibility rules
- Company or client limits
- Employee-level limits

## Amount Calculation

Amounts must be calculated by deterministic code.

Allowed calculation sources:

- Explicit monthly amount
- Explicit maximum duration
- Explicit yearly maximum
- Explicit normalized amount entered during reviewed policy structuring

Disallowed calculation behavior:

- Inferring amount from policy category
- Filling missing durations from similar policies
- Guessing caps from public memory
- Letting an LLM calculate final amounts

## Duplicate-Benefit Rules

Duplicate-benefit compatibility must be represented in data and enforced in code.

Recommended fields:

```json
{
  "duplicate_allowed": ["policy_or_support_type"],
  "duplicate_disallowed": ["policy_or_support_type"],
  "duplicate_rule_evidence_snippets": ["string"]
}
```

If duplicate compatibility is unknown, the safer state is `needs_review`, not allowed.

Policy-level `combination_rules` may declare these source-backed rule types:

- `mutually_exclusive`: the current policy and target policies must not be selected together when the rule applies.
- `requires`: the current policy requires one or more target policies or policy contexts to be present.
- `allowed_with`: the current policy is explicitly allowed with the listed target policies.

Current implementation scope is schema validation only. It validates structure, allowed rule types, target references, reasons, and evidence snippets. It does not perform runtime policy conflict detection, exclusion, total amount aggregation, combination generation, or net-cost optimization.

## Mutually Exclusive Detection

The current conflict detector supports only `mutually_exclusive` rules.

Detection rules:

- Only policies with `review_status == "approved"` are considered.
- Schema validation runs before conflict detection by reusing the `combination_rules` validator.
- Invalid rules are returned as structured schema errors and are not auto-corrected.
- A `mutually_exclusive` rule creates a conflict only when the target policy is also present in the current approved candidate list.
- A one-way declaration such as `A -> B` is enough to report a conflict when both policies are candidates.
- If both `A -> B` and `B -> A` are declared, the conflict is returned once.
- `requires` and `allowed_with` rules are ignored by this detector.
- Evidence snippets are returned exactly as stored in the rule.

This detector does not remove policies, generate valid combinations, sum support amounts, or optimize net cost.

## Requires Detection

The current requirement detector supports only `requires` rules.

Detection rules:

- Only policies with `review_status == "approved"` are considered.
- Schema validation runs before requirement detection by reusing the `combination_rules` validator.
- Invalid rules are returned as structured schema errors and are not auto-corrected.
- A `requires` rule is checked only when its source policy is present in the current approved candidate list.
- Every policy ID listed in `target_policy_ids` is required.
- If any required target policy is missing from the approved candidate list, the rule creates a violation.
- A target policy present with `review_status == "needs_review"` or `review_status == "deprecated"` is treated as missing.
- OR requirements and `requirement_mode` are not supported in the current scope.
- `mutually_exclusive` and `allowed_with` rules are ignored by this detector.
- Evidence snippets are returned exactly as stored in the rule.

This detector does not add missing policies, generate valid combinations, exclude conflicting policies, sum support amounts, or optimize net cost.

## Policy Combination Generation

The current combination generator creates candidate policy subsets and classifies them as valid or rejected.

Generation rules:

- Only policies with `review_status == "approved"` are used as combination candidates.
- Empty combinations are not generated.
- All non-empty subsets of approved candidates are generated up to the safety limit.
- Policy IDs inside each combination are sorted into a stable order.
- Result lists are returned in deterministic order for reproducible tests.
- Duplicate policy IDs in the input return a structured error and are not merged.
- The generator reuses `detect_mutually_exclusive_conflicts` for conflict checks.
- The generator reuses `detect_required_policy_violations` for required-policy checks.
- Combinations with no conflicts and no requires violations are returned in `valid_combinations`.
- Combinations with conflicts or requires violations are returned in `rejected_combinations`.
- Rejection reasons preserve detector details and evidence snippets.
- `allowed_with` rules do not affect generation results in the current scope.
- Policies are not automatically added or removed to satisfy a rule.
- Amount calculation, total support aggregation, best-combination selection, and employer net-cost optimization are not performed.

Safety guard:

- `MAX_COMBINATION_CANDIDATES = 12`
- If approved candidates exceed this count, generation stops and returns a structured error.

## Combination Amount Summarization

The current amount summarizer totals already calculated policy results for valid combinations.

Summarization rules:

- Input combinations must come from `generate_valid_policy_combinations`.
- Policy-level inputs must be standardized calculation results from `normalize_policy_calculation_result`.
- Only `valid_combinations` are summarized.
- Policy results are matched by `policy_id`.
- Every policy in a combination must have `status == "calculated"`.
- A combination is rejected if any policy result is missing, `ineligible`, `calculation_error`, not calculated, or has a null `estimated_total_amount`.
- Duplicate policy calculation results for the same `policy_id` return a structured error and are not merged.
- `total_subsidy_amount` is calculated only by summing each policy's `estimated_total_amount`.
- `total_base_amount` is calculated by summing each policy's `base_amount`.
- `total_bonus_amount` is calculated by summing each policy's `bonus_amount`.
- Conditional bonus amounts are not added a second time because they are already included in `estimated_total_amount`.
- Null amount fields are not converted to zero.
- Policy result order follows each combination's stable `policy_ids` order.
- Calculation steps and evidence snippets are copied from existing policy calculation results and are not regenerated.
- Amount summarization does not perform employer cost calculation, net-cost calculation, ranking, or optimal-combination selection.

## Employer Net Cost Calculation

The current employer net-cost calculator applies only explicitly provided employer cost items to already summarized combinations.

Input boundary:

- The calculator consumes `summarized_combinations` produced by `summarize_combination_amounts`.
- It uses only explicitly supplied `employer_cost_items`.
- It does not inspect policy source text, employee data, policy category, or subsidy amount to create or infer costs.

Calculation formula:

- `total_employer_cost = sum(applied employer_cost_items.amount)`
- `net_employer_cost = total_employer_cost - total_subsidy_amount`

Cost item rules:

- Cost items are never generated by code.
- Costs are never inferred from policy text, employee data, support amount, or policy category.
- If `employer_cost_items` is missing or empty, it is treated as an explicit empty list and `total_employer_cost` is `0`.
- Each cost item must have `cost_id` and numeric non-negative `amount`.
- Null, non-numeric, or negative `amount` values return structured validation errors and are not converted to zero.
- Duplicate `cost_id` values return structured validation errors and are not merged.
- Missing `applies_to_policy_ids` is normalized to an empty list.
- If `applies_to_policy_ids` is empty, the cost applies to every summarized combination.
- If `applies_to_policy_ids` contains policy IDs, every listed policy ID must be present in the combination for the cost to apply.
- Multiple `applies_to_policy_ids` are treated as an AND condition.
- OR cost applicability is not supported in the current scope.
- Non-applicable cost items are ignored for that combination.
- Existing `total_subsidy_amount` is reused and not recalculated.
- Negative `net_employer_cost` is allowed because support may exceed explicit employer costs.
- Net-cost calculation does not choose an optimal combination, rank recommendations, or sort by amount.
- Acceptance verification includes a test-only `employer_net_cost` scenario that compares expected `total_subsidy_amount`, `total_employer_cost`, `net_employer_cost`, and applied `cost_id` values against the existing calculator output.

## Combination Optimization

The optimizer should select a combination that maximizes expected support amount while respecting:

- Employee-level conflicts
- Company-level limits
- Policy-specific duplicate rules
- Employee count limits
- Duration limits
- Approved policy gate

The current code contains a simple amount-sorted selection path. That is acceptable as an initial harness baseline, but it should not be described as full optimization until conflict and limit coverage is complete.


## MVP Separation

Recommendation results must remain scoped to one of these contexts:

- A single general company
- A single labor partner's selected client company

The MVP must not recommend based on data shared between general companies and labor partners.
