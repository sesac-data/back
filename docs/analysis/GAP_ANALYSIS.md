# Incentdoc v2 Gap Analysis

## 1. 현재 기술 스택 관점의 갭

현재 저장소는 Streamlit 기반 기존 추천 앱과 React/Vite 기반 v2 목업성 프론트가 공존한다. v2 목업은 운영 SaaS 구조를 요구하지만, 실제 기능은 Streamlit 단일 흐름과 정책 추천 엔진에 더 가깝다.

- 기존 기능 중심: 파일 업로드, 정책 분석, 추천 결과
- v2 요구사항 중심: 사용자 유형별 운영 허브, 온보딩, 직원/고객사/서류/일정/알림 관리
- 가장 큰 갭: 프론트, 백엔드, DB가 하나의 제품 아키텍처로 묶여 있지 않음

## 2. 현재 디렉터리 구조 관점의 갭

현재 구조에는 `pages/`, `services/`, `database/`, `frontend/`가 있지만 v2 도메인 기준의 경계는 없다.

필요하지만 없는 구조 예시는 다음과 같다.

```text
accounts / auth
organizations / companies / labor_partners
clients
employees
leave_events
documents
deadlines
notifications
recommendations
```

현 구조의 서비스들은 정책 엔진 중심이고, v2의 운영 데이터 흐름을 표현하는 모듈은 부족하다.

## 3. 이미 구현된 기능과 v2 요구사항 매핑

| v2 요구사항 | 현재 상태 | 갭 |
|---|---|---|
| 랜딩/로그인/사용자 유형 선택 | React 정적 화면 일부 | 실제 인증/저장 없음 |
| 일반기업 온보딩 | React 정적 Wizard | 저장, 검증, 재진입 없음 |
| 노무법인 온보딩 | React 정적 Wizard | 첫 고객사/직원 저장 없음 |
| 기업관리 허브 | React 정적 허브, Streamlit 추천 결과 | 실제 집계 데이터 없음 |
| 일반기업 직원 관리 | React 정적 직원 카드 | CRUD/업로드/휴직 기간 저장 없음 |
| 노무법인 고객사 관리 | React 정적 고객사 목록 | 고객사 CRUD/권한 없음 |
| 서류/일정 관리 | React/목업 화면 | 제출서류 상태/마감 계산 미구현 |
| 대체인력 관리 | React/목업 화면 | 실제 추천/공고/워크넷 연동 없음 |
| 뉴스레터 발송 | React/목업 화면 | 발송 대상/초안/전송 없음 |
| 정책 추천 | Python 서비스 일부 | v2 데이터 모델과 미연결 |

## 4. 미구현 기능

- 일반기업 신규 가입: 사업자번호, 회사정보, 담당자, 계정 생성
- 노무법인 신규 가입: 사업자 검증, 법인정보, 대표 노무사, 계정 생성
- 사용자 유형별 로그인 이후 라우팅
- 기존 사용자 온보딩 skip 판별
- 온보딩 저장 데이터와 상시 관리 화면의 동일 데이터 재사용
- 일반기업 직원 관리 하위화면
- 노무법인 고객사 기준 직원 관리 하위화면
- 서류 제출/검수 상태, 수정 요청, 승인/접수
- 마감 알림과 신청 일정 계산
- 대체인력 공고 초안/업무부담/추가 수급액 확인
- 뉴스레터 주제 추천/초안 생성/대상 선택/일괄 발송
- 파트너 설정의 수수료/계좌/알림/플랜 관리

## 5. 목업과 실제 코드의 차이

목업과 이미지 기준 핵심 원칙은 다음이다.

- 일반기업과 노무법인은 독립 플로우로 운영한다.
- 온보딩은 최초 1회 입력 가이드다.
- 온보딩 이후에는 각 운영 메뉴에서 동일 데이터를 상시 수정한다.
- 직원 관리는 독립 메뉴처럼 보일 수 있지만 일반기업은 `기업 관리`, 노무법인은 `고객사 관리`의 하위화면이다.

실제 코드와의 차이는 다음이다.

- Streamlit 앱은 위 IA가 아니라 업로드 -> 확인 -> 추천 -> 신청서 흐름이다.
- React 앱은 IA를 흉내 내지만 실제 상태 저장, API, DB, 권한이 없다.
- 목업의 두 플로우는 데이터 소유 구조가 다르지만, 실제 DB에는 그 구조가 없다.
- 정책 추천 엔진은 존재하나 화면의 직원/휴직/고객사 데이터와 연결되어 있지 않다.

## 6. 일반기업과 노무법인 플로우 분리 여부

현재 분리 수준은 낮다.

- 화면 목업 수준: React `role` state로 일반기업/노무법인 메뉴 분기 존재
- 실제 기능 수준: 미분리
- DB 수준: 미분리
- 권한 수준: 미분리
- 정책 엔진 입력 수준: 미분리

따라서 v2 구현의 첫 구조적 갭은 `user_type`과 조직 컨텍스트의 부재다.

## 7. 정책 문서 추출, 조건 판별, 지원금 계산, 조합 추천 상태

| 기능 | 현재 구현 | 주요 갭 |
|---|---|---|
| 정책 문서 추출 | OpenAI로 원문을 JSON schema로 변환 | 운영 승인, 검수, 재추출 관리 부족 |
| 조건 판별 | child_age, employee_count, weekly_work_hours 등 일부 지원 | policy JSON의 condition type과 완전 일치하지 않음 |
| 지원금 계산 | 월액 x 기간, 연 한도, 정규화 금액 사용 | 날짜 기반 기간 계산, 인원 한도, 회사 단위 한도 미흡 |
| 조합 추천 | 중복 정책 제외 후 금액 기준 선택 | 최적 조합, 정책별 상호 배타 규칙, 회사/직원 단위 혼합 한도 부족 |

## 8. 가장 위험한 부분

1. 데이터 모델 없이 화면을 먼저 구현하는 것
2. 일반기업과 노무법인 데이터가 섞일 수 있는 권한/소유권 부재
3. 정책 엔진 결과의 신뢰성 부족
4. React 목업과 Streamlit 실제 앱의 제품 경로 불일치
5. 하드코딩 DB URL과 로컬 전제
6. 정책 조건 type 불일치로 인한 무음 실패

## 9. 의존성을 고려한 구현 순서

1. 제품 실행 표면 확정
2. 계정과 `user_type` 도입
3. 일반기업/노무법인/고객사/직원 기본 모델 도입
4. 온보딩 저장 흐름 구현
5. 사용자 유형별 허브 라우팅 구현
6. 직원 관리 CRUD 구현
7. 휴직 유형, 시작일, 종료일 저장 및 중복 기간 검증
8. 정책 엔진 입력 어댑터 구현
9. 추천 결과 저장/조회 구현
10. 서류/일정 관리, 대체인력, 뉴스레터 순차 구현

## 10. 첫 번째로 구현해야 할 최소 기능

첫 최소 기능은 "사용자 유형별 온보딩 데이터 저장 후 허브 진입"이다.

범위는 작게 잡아야 한다.

- 로그인은 임시 세션 또는 기존 데모 진입으로 대체 가능
- 일반기업: 회사 기본정보 1건, 직원 1명 이상 저장
- 노무법인: 법인 기본정보 1건, 고객사 1건, 고객사 직원 1명 이상 저장
- 저장 후 사용자 유형별 허브에서 해당 데이터 수만 표시

이 기능이 완성되면 이후 직원 관리, 정책 추천, 서류 일정이 모두 같은 데이터 위에 올라갈 수 있다.
