# Incentdoc v2 Implementation Order

## 1. 현재 기술 스택 전제

새 라이브러리 설치 없이 진행한다는 조건에서는 기존 스택을 활용하는 것이 현실적이다.

- 프론트엔드: 기존 React/Vite/lucide-react
- 기존 기능 검증용 앱: Streamlit
- 정책 엔진: 기존 Python `services/*`
- 저장소: 기존 SQLAlchemy/PostgreSQL 코드 또는 초기에는 동일 인터페이스의 단순 저장 방식
- 정책 데이터: 기존 `data/policy_json/*.json`

우선순위는 "정책 엔진 고도화"가 아니라 "v2 제품 데이터 흐름의 뼈대"다.

## 2. 현재 디렉터리 구조 기준 작업 위치

초기 구현은 기존 구조를 크게 바꾸지 않는 범위에서 시작하는 것이 안전하다.

```text
match_agent_v0.8/match_agent_v0.8/
  frontend/src/
    App.jsx
    styles.css
  services/
    existing policy/recommendation services
  database/
    existing SQLAlchemy base/models/crud
```

새 기능을 붙일 때도 먼저 작은 단위로 추가해야 한다.

- 계정/조직 모델
- 온보딩 저장 API 또는 저장 서비스
- 직원/고객사 데이터 접근 서비스
- 정책 엔진 입력 변환 서비스

## 3. 이미 구현된 기능을 활용하는 순서

1. React 화면의 역할 분기와 메뉴 구성을 v2 IA 검증용으로 활용한다.
2. 기존 정책 JSON과 추천 서비스를 엔진 후보로 유지한다.
3. Streamlit 업로드/추천 흐름은 참고 구현으로 두고, v2 운영 화면과 직접 섞기 전에 입력/출력 계약을 분리한다.
4. DB의 `RecommendationLog`는 추천 결과 저장의 참고 모델로 활용하되, 조직/직원 FK가 없는 현재 형태는 그대로 확장하지 않는다.

## 4. 미구현 기능의 의존성

```text
사용자 유형/조직 컨텍스트
  -> 온보딩 저장
    -> 직원/고객사 관리
      -> 휴직 이벤트 관리
        -> 정책 추천 입력 생성
          -> 지원금 계산/추천 결과 저장
            -> 서류/일정 관리
              -> 대체인력/뉴스레터/알림
```

앞 단계가 없으면 뒤 단계는 정적 화면 또는 임시 데이터에 머물 가능성이 높다.

## 5. 목업과 실제 코드 차이를 줄이는 단계

1. React의 `role` 상태명을 제품 스펙의 `general_company`, `labor_partner`와 맞춘다.
2. 랜딩/로그인/온보딩/허브의 화면 이동 규칙을 명확히 한다.
3. 온보딩 완료 여부를 저장한다.
4. 일반기업 메뉴와 노무법인 메뉴를 같은 컴포넌트 조건 분기가 아니라 데이터 소유 모델 기준으로 분리한다.
5. 직원 관리를 일반기업은 회사 하위, 노무법인은 고객사 하위 컨텍스트로 강제한다.

## 6. 일반기업과 노무법인 플로우 분리 구현 순서

1. `user_type` 정의
   - `general_company`
   - `labor_partner`
2. 사용자별 기본 조직 연결
   - 일반기업 사용자는 `company_id`
   - 노무법인 사용자는 `labor_partner_id`
3. 노무법인의 고객사 모델 추가
   - 고객사는 노무법인에 속한다.
   - 고객사의 직원은 고객사에 속한다.
4. 일반기업 직원 모델 추가
   - 직원은 일반기업 회사에 속한다.
5. 라우팅과 메뉴 표시를 `user_type` 기준으로 제한한다.
6. 추후 MVP 범위 밖인 일반기업-노무법인 연결/매칭은 만들지 않는다.

## 7. 정책 엔진 기능 연결 순서

1. 현재 정책 JSON condition type 목록을 전수 확인한다.
2. `condition_evaluator.py`가 지원하지 않는 type을 목록화한다.
3. 직원/휴직/회사 데이터를 정책 엔진 입력 형태로 변환하는 adapter를 만든다.
4. 날짜 기반 휴직 기간, 근로시간, 자녀 나이 등 최소 조건만 먼저 연결한다.
5. 계산 결과를 화면에 바로 뿌리기 전에 추천 로그로 저장한다.
6. 조합 추천은 초기에는 기존 금액 기준 선택을 유지하고, 정책 충돌 규칙을 테스트로 잠근 뒤 확장한다.

## 8. 가장 위험한 부분을 줄이는 순서

1. 데이터 소유권 먼저 고정
2. 정책 엔진 입력/출력 계약 고정
3. condition type 불일치 테스트 추가
4. 추천 결과에 "판별 불가"와 "근거 부족" 상태를 명시
5. DB 접속정보를 환경변수 기반으로 전환
6. React와 Streamlit 중 실제 제품 경로를 결정

## 9. 전체 구현 순서

| 순서 | 작업 | 이유 |
|---:|---|---|
| 1 | `user_type`과 조직 컨텍스트 정의 | 모든 플로우의 분기 기준 |
| 2 | 일반기업/노무법인/고객사/직원 최소 모델 | 데이터 소유권 확정 |
| 3 | 온보딩 저장 | 첫 진입 플로우 완성 |
| 4 | 사용자 유형별 허브 | v2 IA의 중심 화면 |
| 5 | 직원 관리 | 추천 엔진의 핵심 입력 |
| 6 | 휴직 유형/기간 관리 | 정책 조건 판별의 핵심 입력 |
| 7 | 정책 엔진 입력 adapter | 기존 엔진 재사용 연결점 |
| 8 | 추천 결과 저장/조회 | 운영 화면과 이력 기반 |
| 9 | 서류/일정 관리 | 추천 이후 업무 |
| 10 | 대체인력 관리 | 일반기업 확장 업무 |
| 11 | 뉴스레터 발송 | 노무법인 확장 업무 |
| 12 | 알림/권한/감사 로그 | 운영 안정화 |

## 10. 첫 번째로 구현해야 할 최소 기능

첫 번째 기능은 "사용자 유형별 최소 온보딩 저장과 허브 진입"이다.

### 목표

- 일반기업과 노무법인이 같은 랜딩에서 시작한다.
- 사용자가 유형을 선택한다.
- 신규 사용자는 최초 1회 온보딩을 거친다.
- 온보딩 데이터가 저장된다.
- 완료 후 사용자 유형별 허브로 이동한다.

### 최소 입력

일반기업:

- 회사명
- 사업자번호
- 담당자 이름/이메일
- 직원 1명 이름
- 휴직 유형
- 시작일/종료일

노무법인:

- 법인명
- 사업자번호
- 담당자 이름/이메일
- 첫 고객사명
- 고객사 사업자번호
- 고객사 직원 1명 이름
- 휴직 유형
- 시작일/종료일

### 완료 기준

- 일반기업 허브에서 회사명, 직원 수, 휴직 대상 수가 보인다.
- 노무법인 허브에서 법인명, 고객사 수, 직원 수가 보인다.
- 일반기업 직원은 일반기업 회사에만 속한다.
- 노무법인 직원은 선택 고객사에만 속한다.
- 일반기업과 노무법인 데이터는 자동 공유되지 않는다.

이 최소 기능이 끝나야 정책 추천, 서류 일정, 대체인력, 뉴스레터 기능을 실제 데이터 위에 올릴 수 있다.
